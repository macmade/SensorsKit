Edits that touch the same method are grouped into a single milestone to avoid conflicting changes across commits. Every milestone keeps the SensorsKit.xcodeproj and the Swift Testing suite in sync and is validated with xcodebuild in both Debug and Release.

Medium 1
Goal

  • Add/extend a test in SensorsTests.swift asserting the published order is stable and correctly sorted across passes.
  • Build & test via xcodebuild (Debug + Release); keep the xcodeproj/tests in sync.

Implementation notes
  • Added an internal static func sorted( _: [ SensorHistoryData ] ) -> [ SensorHistoryData ] to Sensors that orders histories by the composite key (source.rawValue, kind.rawValue, name) using Swift's lexicographic tuple < (all three components are Comparable; String comparison is non-localized Unicode-scalar, which keeps it deterministic across runs).
  • readSensors() now publishes Sensors.sorted( [ SensorHistoryData ]( sensors.values ) ) instead of the unordered dictionary values. The sort is computed on the background polling thread and only the self.data assignment stays on the main queue.
  • Confirmed the key is a total order for the published set: the sensors dictionary keys ("IOHID.%llu.%@" / "SMC.%llu.%@" built from source, kind.rawValue and name) are a bijection with the (source, kind, name) triple, so no two published values share a key — the order is unique and does not depend on sorted's stability.
  • Chosen approach (agreed with the owner): the helper is internal, not public, and the test reaches it via @testable import SensorsKit — the public API is unchanged.
  • TDD: added the failing test sortedProducesDeterministicOrder() first (compile error — type 'Sensors' has no member 'sorted'), then implemented the helper until it passed.
  • CI adjustment: the Release configuration builds the framework with ENABLE_TESTABILITY = NO, so @testable import cannot resolve the module in the Release test action. Per the owner's explicit direction, the Release matrix entry in .github/workflows/ci-mac.yaml was changed to test: 0 (it still runs build, analyze and info); Debug keeps test: 1, where @testable works. The Release build action does not compile the test bundle (the scheme's build action contains only the framework), so it is unaffected.
  • Verification: Debug xcodebuild test → 14/14 pass (including the new test); Release xcodebuild build → BUILD SUCCEEDED.
  • Automated review loop: 1 round; converged with no Critical/High/Medium findings.
  • Follow-up (owner review): the name comparison was changed from plain code-point < to a case-insensitive, numeric-aware comparison pinned to a fixed en_US locale (name.compare( _, options: [ .numeric, .caseInsensitive ], range: nil, locale: Sensors.sortLocale )), cached in a private static let sortLocale. This orders names the way a person reads them ("Core 2" before "Core 10"; case ignored) while staying independent of the host's locale. When the locale-aware comparison returns .orderedSame for two distinct names (e.g. "TC0P" vs "tc0p", or numerically equal forms), a plain code-point tie-break preserves a total order, so full determinism is retained. Caveat accepted: locale collation depends on the OS's ICU tables, so the order is identical across machines on the same OS/ICU version rather than immutable across OS upgrades — a non-issue for the ASCII SMC keys and simple IOHID names in practice.
  • TDD (follow-up): added sortedUsesCaseInsensitiveNaturalOrdering() (fails under code-point ordering — "Core 10"/"Zebra" misordered — passes after the change) and sortedIsDeterministicForCollationEquivalentNames() (guards the tie-break). Verified: Debug xcodebuild test → 16/16 pass; Release xcodebuild build → BUILD SUCCEEDED.

M2 — Polling loop lifecycle & threading rework

Medium 2 Low 1 Low 3 Low 5
Goal

Rework the polling loop so a Sensors object no longer retains itself for the process lifetime, can be reclaimed and stopped on deallocation, does not indefinitely occupy a shared GCD worker thread, tears down promptly on stop(), and never silently drops a completion closure. This is the plan's largest, riskiest change and is intentionally the one behavioral rework; all four findings here touch run() / stop() / the timer, so they are done together to avoid conflicting edits.

Tasks
  • Break self-retention (Medium 2): capture self weakly in the loop dispatch and in the timer closure ({ [weak self] _ in self?.readSensors() }), guarding for nil, so the running loop no longer keeps the object alive.
  • Dedicated thread (Low 1): replace the borrowed .background global-queue thread with a dedicated Thread that owns the run loop for the object's lifetime, so pooled libdispatch workers are not held indefinitely.
  • Deinit-driven teardown (Medium 2): add a deinit that requests the loop to end (sets shouldRun = false and wakes the run loop), so dropping the object without stop() stops polling and deallocates it rather than leaking.
  • Prompt stop (Low 1): wake/stop the run loop from stop() (and deinit) instead of relying on the 5 s run(until:) window, so teardown latency is minimal.
  • Remove redundant timer add (Low 3): drop the duplicate RunLoop.current.add(timer, forMode: .default) — scheduledTimer already schedules it — or restructure to a single explicit Timer(timeInterval:) + one add(...) on the dedicated thread's run loop.
  • Completion on repeat stop (Low 5): when stop() is called while already stopped, invoke the supplied completion immediately (documenting the semantics) instead of discarding it, so an awaiting caller never hangs.
  • Ensure thread-safe access to shouldRun / completion across the dedicated thread and calling threads (reuse Synchronizable or an equivalent guard).
  • Update the SwiftDoc on init(), stop(completion:), run() and the new deinit to reflect the new lifetime/threading contract.
  • Tests (SensorsTests.swift): stop() invokes its completion exactly once; a second stop() still invokes its completion; releasing a Sensors without calling stop() lets it deallocate (assert a weak reference becomes nil); stop teardown is prompt.
  • Build & test via xcodebuild (Debug + Release); keep the xcodeproj/tests in sync.
