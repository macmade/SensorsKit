Overview

This plan turns the 2026-07-12 SensorsKit code review into a sequence of small, independently reviewable and committable milestones. The review found no critical or high-severity defects; the work here addresses all eleven findings — two Medium, five Low and four Info — plus a README typo.

The milestones are ordered from the most user-visible correctness issue (nondeterministic ordering of the published data) through the most substantial change (a full rework of the polling loop's lifecycle and threading, which the user chose over a document-only approach), and finish with low-risk readability, robustness and documentation touch-ups. Edits that touch the same method are grouped into a single milestone to avoid conflicting changes across commits. Every milestone keeps the SensorsKit.xcodeproj and the Swift Testing suite in sync and is validated with xcodebuild in both Debug and Release.

Goal

Make the KVO-observable Sensors.data array have a stable, deterministic element order so a consumer binding it to a table or list no longer sees rows shuffle arbitrarily between polling passes, and so tests and UI are reproducible.

Tasks

  • Consider extracting the ordering into a small private helper so it can be unit-tested directly without hardware.
  • Add/extend a test in SensorsTests.swift asserting the published order is stable and correctly sorted across passes.
  • Build & test via xcodebuild (Debug + Release); keep the xcodeproj/tests in sync.

Implementation notes

  • Follow-up (owner review): the name comparison was changed from plain code-point < to a case-insensitive, numeric-aware comparison pinned to a fixed en_US locale (name.compare( _, options: [ .numeric, .caseInsensitive ], range: nil, locale: Sensors.sortLocale )), cached in a private static let sortLocale. This orders names the way a person reads them ("Core 2" before "Core 10"; case ignored) while staying independent of the host's locale. When the locale-aware comparison returns .orderedSame for two distinct names (e.g. "TC0P" vs "tc0p", or numerically equal forms), a plain code-point tie-break preserves a total order, so full determinism is retained. Caveat accepted: locale collation depends on the OS's ICU tables, so the order is identical across machines on the same OS/ICU version rather than immutable across OS upgrades — a non-issue for the ASCII SMC keys and simple IOHID names in practice.
  • TDD (follow-up): added sortedUsesCaseInsensitiveNaturalOrdering() (fails under code-point ordering — "Core 10"/"Zebra" misordered — passes after the change) and sortedIsDeterministicForCollationEquivalentNames() (guards the tie-break). M2 — Polling loop lifecycle & threading rework

Medium 2 Low 1 Low 3 Low 5
Goal

Rework the polling loop so a Sensors object no longer retains itself for the process lifetime, can be reclaimed and stopped on deallocation, does not indefinitely occupy a shared GCD worker thread, tears down promptly on stop(), and never silently drops a completion closure. This is the plan's largest, riskiest change and is intentionally the one behavioral rework; all four findings here touch run() / stop() / the timer, so they are done together to avoid conflicting edits.

Tasks
  • Break self-retention (Medium 2): capture self weakly in the loop dispatch and in the timer closure ({ [weak self] _ in self?.readSensors() }), guarding for nil, so the running loop no longer keeps the object alive.
  • Dedicated thread (Low 1): replace the borrowed .background global-queue thread with a dedicated Thread that owns the run loop for the object's lifetime, so pooled libdispatch workers are not held indefinitely.
  • Deinit-driven teardown (Medium 2): add a deinit that requests the loop to end (sets shouldRun = false and wakes the run loop), so dropping the object without stop() stops polling and deallocates it rather than leaking.
  • Prompt stop (Low 1): wake/stop the run loop from stop() (and deinit) instead of relying on the 5 s run(until:) window, so teardown latency is minimal.
  • Remove redundant timer add (Low 3): drop the duplicate RunLoop.current.add(timer, forMode: .default) — scheduledTimer already schedules it — or restructure to a single explicit Timer(timeInterval:) + one add(...) on the dedicated thread's run loop.
  • Completion on repeat stop (Low 5): when stop() is called while already stopped, invoke the supplied completion immediately (documenting the semantics) instead of discarding it, so an awaiting caller never hangs.
  • Ensure thread-safe access to shouldRun / completion across the dedicated thread and calling threads (reuse Synchronizable or an equivalent guard).
  • Update the SwiftDoc on init(), stop(completion:), run() and the new deinit to reflect the new lifetime/threading contract.
  • Tests (SensorsTests.swift): stop() invokes its completion exactly once; a second stop() still invokes its completion; releasing a Sensors without calling stop() lets it deallocate (assert a weak reference becomes nil); stop teardown is prompt.
  • Build & test via xcodebuild (Debug + Release); keep the xcodeproj/tests in sync.
Status   Completed
Completed   2026-07-13
Implementation notes
  • Mechanism (agreed with the owner): "Option A" — keep a run loop + Timer, but on a dedicated Thread. Discussed the trade-offs against an NSCondition-based wait ("Option B"); the owner chose Option A to keep the Timer-driven fixed-rate cadence.
  • Verified the run-loop primitives empirically before implementing (small standalone programs): with returnAfterSourceHandled = false, timer fires do not return CFRunLoopRunInMode (only stop/timeout does); CFRunLoopStop from another thread wakes a blocked run immediately; and a CFRunLoopStop issued before the run call is lost (does not persist).
  • No self-retention (Medium 2): init spawns a dedicated Thread whose block captures [weak self], and the polling loop lives inline in the block (not behind a single self.run() call, which would hold a strong self across the whole loop). The Timer closure also captures [weak self]. So no strong self is held across the blocking CFRunLoopRunInMode, and the object can deallocate while the thread is blocked.
  • Deinit teardown (Medium 2): deinit drains pending completions (idempotently), clears shouldRun, wakes the run loop, and invokes the completions. No escaping closure in the deinit path captures self (the DispatchQueue.main.async { self.data = [] } is confined to the running-loop path, never deinit).
  • Prompt stop (Low 1): stop()/deinit call CFRunLoopStop for the instant common-case wake, backed by a finite pollInterval (1 s) limit on CFRunLoopRunInMode so a lost pre-run stop can delay teardown by at most one interval instead of hanging.
  • Dedicated thread (Low 1): a Thread owns the run loop for the object's lifetime instead of borrowing a pooled DispatchQueue.global() worker. QoS restored to .background to match the original priority (see Automated review).
  • Single timer add (Low 3): the timer is created with Timer(timeInterval:repeats:) and added once via RunLoop.add(_:forMode:) — the redundant second add is gone.
  • Completion on repeat stop (Low 5): completions are queued in an array and drained exactly once via a stopped flag; a stop() after the loop has ended invokes its completion immediately. So a second stop() no longer silently discards its completion.
  • Thread-safety: Sensors now conforms to Synchronizable; shouldRun/stopped/completions/runLoop are only touched inside synchronized. Completions are always invoked outside the lock to avoid re-entrancy deadlock.
  • TDD: wrote the lifecycle tests first — secondStopStillInvokesCompletion (hung under the old code), deallocatesWhenReleasedWithoutStop (weak ref never cleared under the old code), and a promptness test — watched them fail, then implemented. Discovered that stop latency is bounded by the in-flight readSensors() pass (SMC.readAllKeys() ≈ 0.92 s for ~1980 keys, so a full pass ≈ 1–2 s), which is hardware-dependent; replaced the flaky wall-clock stopIsPrompt with a deterministic repeatStopCompletesPromptly that bounds only the already-stopped immediate path.
  • Verification: Debug xcodebuild test → 20/20 pass, stable across repeated runs; Release xcodebuild build → BUILD SUCCEEDED.
  • Automated review loop: 1 round; converged with no Critical/High/Medium findings.
