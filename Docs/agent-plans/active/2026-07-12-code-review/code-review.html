SensorsKit — Code Review

Generated: 2026-07-12  ·  Scope: Whole repository (primary target SensorsKit/; vendored Submodules/ reviewed only where the main code depends on them)

Summary

SensorsKit is a small, focused macOS framework (three Swift files) that polls hardware sensors through the IOHID and SMC subsystems once per second and publishes an observable, key-value-observing history of readings. The code is clean, well-documented, and the thread-safety model — a background poller that mutates each SensorHistoryData through a Synchronizable lock while main-thread observers read the same objects — is sound and matches the intent recorded in the recent "Updating data on main thread" commit. The Swift Testing suite is reasonable and CI builds and tests both Debug and Release.

No critical or high-severity defects were found. The most consequential issues are two mediums in Sensors.swift: the published data array is rebuilt from an unordered dictionary each pass, so its element order is nondeterministic and will visibly shuffle in any list-based UI; and the polling object retains itself for the process lifetime, so a consumer that drops it without calling stop() leaks the object and leaves a 1 Hz hardware-polling loop running forever on a borrowed GCD worker thread. The remaining findings are lower-severity robustness, correctness-of-classification, and readability points.

Findings

Medium correctness API / UX

Published data has nondeterministic element order

SensorsKit/Sensors.swift:135–141

Each polling pass republishes the histories with self.data = [ SensorHistoryData ]( sensors.values ), where sensors is a [String: SensorHistoryData]. Swift's Dictionary.values has an unspecified iteration order, and that order changes as keys are inserted — which happens whenever a new sensor first appears during the run. A consumer that binds data to a table or list (the natural use of a KVO-observable array) will see rows reorder arbitrarily from one second to the next, and cannot rely on index stability across updates.

Recommendation

Publish a deterministically ordered array, e.g. sort by a stable composite key before assigning: self.data = sensors.values.sorted { ($0.source.rawValue, $0.kind.rawValue, $0.name) < ($1.source.rawValue, $1.kind.rawValue, $1.name) }. This also makes the tests and any UI deterministic.

Medium stability resource leak

Object retains itself for its lifetime; forgetting stop() leaks it and never halts polling

SensorsKit/Sensors.swift:62–124 (no deinit)

init() dispatches self.run() onto a global queue, and that escaping closure — together with the repeating timer's { _ in self.readSensors() } closure — holds a strong reference to self for as long as run() is executing. run() only returns once shouldRun becomes false, which happens exclusively inside stop(). There is no deinit, and none is reachable while the loop runs.

Consequently, if a caller creates a Sensors and simply releases its last reference without calling stop(), ARC cannot deallocate it: the object leaks and its 1 Hz polling loop (reading all SMC keys plus every IOHID sensor each second) keeps running for the remainder of the process. For a reusable framework this is an easy footgun to hit.

Recommendation

Document explicitly that stop() is mandatory for reclamation, and/or break the self-retention so the object can stop on deallocation — e.g. drive the loop from a weak reference and add a deinit that requests the loop to end. At minimum, note in the class docs that dropping the object without stop() leaks it.

Low performance threading

Run loop is driven on a shared GCD worker thread; stop() latency up to ~5 s

SensorsKit/Sensors.swift:67–70, 108–111

The polling loop runs RunLoop.current.run(until:) on a thread borrowed from the .background global concurrent queue, and holds that thread for the object's entire lifetime. Occupying a pooled libdispatch worker thread indefinitely is discouraged; several concurrent Sensors instances could starve the background pool. Separately, the loop only re-checks shouldRun every 5 seconds (Date(timeIntervalSinceNow: 5)), so after stop() the timer keeps firing readSensors() and the completion is delayed by up to ~5 s.

Recommendation

Prefer a dedicated Thread for a long-lived run loop rather than a GCD global-queue thread, and either use a shorter run(until:) window or explicitly wake/stop the run loop from stop() so teardown is prompt.

Low readability

Swapped local variable names in the min / max getters

SensorsKit/SensorHistoryData.swift:112–140

The behavior is correct (the tests confirm min returns the smallest and max the largest), but the local bindings are inverted: the min getter binds self.data.min() to a variable named max and returns it, while the max getter binds self.data.max() to a variable named min. Reading return NSNumber( floatLiteral: max ) inside the property literally named min invites a future "fix" that would actually introduce a bug.

Recommendation

Rename the local bindings to match the property (or bind directly, e.g. guard let value = self.data.min() else { return nil }).

Low correctness

Redundant re-scheduling of the timer on the run loop

SensorsKit/Sensors.swift:100–106

Timer.scheduledTimer(withTimeInterval:repeats:) already creates the timer and adds it to the current run loop in .default mode. The following RunLoop.current.add(timer, forMode: .default) adds the same timer to the same run loop and mode again. A timer belongs to a single run loop; the second add is redundant and misleading (it reads as if it were required for the timer to fire).

Recommendation

Drop the explicit RunLoop.current.add(...), or switch to Timer(timeInterval:...) + a single explicit add(...) if manual scheduling is intended.

Low correctness domain

SMC keys are classified purely by first-letter prefix (T/V/I)

SensorsKit/Sensors.swift:156–163

readSMCSensors() records every key whose name starts with T, V or I and labels it thermal, voltage or current respectively. The SMC keyspace is not that clean: not every T…/V…/I… key is a temperature, voltage or current reading, and non-sensor keys that happen to decode to a Double/Float32 will be surfaced and mislabeled. Keys outside those three prefixes are silently dropped even if they are sensors. This is an inherent heuristic, but worth flagging as a known source of false positives/negatives in the exposed data.

Recommendation

If accuracy matters, gate on a curated key allow-list or on the SMC value type in addition to the name prefix. Otherwise, document that the classification is a best-effort heuristic.

Low API

stop(completion:) silently drops the completion on a repeat call

SensorsKit/Sensors.swift:81–90

If stop() is called a second time (or after the loop has already stopped), the early return means the newly supplied completion is never invoked and is silently discarded. A caller awaiting that completion — e.g. the withCheckedContinuation pattern used in the tests — would hang forever if it happened to be the second stop.

Recommendation

When already stopped, invoke the supplied completion immediately (documenting the semantics), rather than dropping it.

Info KVO

Manual KVO is fired asynchronously and may coalesce intermediate values

SensorsKit/SensorHistoryData.swift:191–202

add(value:) posts willChangeValue/didChangeValue for the four derived keys inside a DispatchQueue.main.async block, after the underlying data has already been updated. When samples arrive faster than main drains, several notifications queue up and each observer re-reads the current (latest) state, so intermediate values may never be observed. This is acceptable for a UI that only wants the current value, but the notification does not correspond to a single discrete change — worth being aware of if a consumer expects one-notification-per-sample.

Info robustness

Synchronizable does not release the lock via defer

SensorsKit/Synchronizable.swift:60–85

Both helpers call objc_sync_enter, run the closure, then objc_sync_exit. This is safe today because the closure type is non-throwing, so it cannot exit early between enter and exit. If the API ever gains a throwing/rethrowing variant, an early exit would leave the lock held. Using defer { objc_sync_exit(self) } would make the release unconditional and future-proof.

Info consistency

description reads min and max under separate locks

SensorsKit/SensorHistoryData.swift:208–214

The description override reads self.min and self.max in two separate synchronized acquisitions, so a concurrent add between them could make the printed min/max reflect slightly different states. Purely cosmetic (debug output), noted for completeness.

Strengths

  • Documentation. Every public and private type and member carries a clear SwiftDoc comment, and the intent (thread model, loading state machine, lifecycle) is explained rather than merely restated.
  • Thread-safety model is coherent. Per-sensor mutation and reads funnel through Synchronizable, and data/loading are published on the main queue — consistent with the deliberate "update on main thread" fix in the history.
  • Tests are honest about nondeterminism. SensorsTests avoids asserting on hardware-dependent readings and instead exercises the observable state machine; SynchronizableTests includes a real 10k-iteration concurrency check.
  • CI builds, analyzes, and tests both Debug and Release with recursive submodules.
  • Conventions (early returns, let over var, no force-unwraps, explicit public, one type per file) are followed throughout the primary target.
